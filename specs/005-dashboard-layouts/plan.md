# Implementation Plan: Dashboard Layouts API

**Branch**: `005-dashboard-layouts` | **Date**: 2026-03-24 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/005-dashboard-layouts/spec.md`

## Summary

Implement a full CRUD API for dashboard layout configurations per tenant. Each layout stores a named set of widgets with grid positions as a JSONB document. The API honors the consumer-driven Pact contract from `embolsadora-frontend`, exposing endpoints under `/api/tenants/{tenantId}/dashboard-layouts` with JWT Bearer authentication. Key business rules: max 3 layouts per tenant, unique names per tenant, and cannot delete the last layout.

## Key Design Decisions

| Decision | Rationale | Trade-offs |
|----------|-----------|-----------|
| **JSONB for widgets** | Widgets always read/written as a unit; no cross-layout widget queries needed | Full array replacement on update; no partial patch |
| **Reuse `/api/tenants` group** | Existing middleware handles JWT + tenant resolution from path | None — identical auth model to edge devices |
| **Partial unique index for name** | `(tenant_id, name) WHERE deleted_at IS NULL` — allows reuse of deleted names | Requires partial index syntax; not supported by all ORMs (not relevant in pgx raw SQL) |
| **Transactional limit check** | COUNT + INSERT in a transaction prevents race conditions | Slightly more complex service code |
| **Soft delete** | Consistent with users table; preserves audit history | `deleted_at IS NULL` filter required on all queries |
| **UUID primary key** | Project standard; Pact accepts any string (type-only matching rule) | Pact examples use timestamp slugs but any string satisfies the contract |

## I. Architecture & Design

### Architectural Alignment

- **Surface**: Registers into existing Gin group `/api/tenants/:tenantId` — same auth model as edge devices
- **Authentication**: JWT Bearer via existing `apimw.JWTAuth()`; tenant resolved via `apimw.ResolveTenantFromPath(db)`
- **Tenant Isolation**: All DB queries filter by `tenant_id` from context (`platform.TenantID`)
- **Layer Pattern**: `transport → app → domain ← repo`

### Route Registration

```go
// internal/routes/url_mappings.go — added to existing tenantsGroup
dashboardLayoutsRepo := dashboardLayoutsRepo.NewPostgresRepository(db)
dashboardLayoutsService := dashboardLayoutsApp.NewService(dashboardLayoutsRepo, logger)
dashboardLayoutsHandler.RegisterRoutes(tenantsGroup, dashboardLayoutsService)
```

### Routes

| Method | Path | Handler | Description |
|--------|------|---------|-------------|
| GET | `/api/tenants/:tenantId/dashboard-layouts` | `ListLayouts` | List all layouts for tenant |
| POST | `/api/tenants/:tenantId/dashboard-layouts` | `CreateLayout` | Create new layout |
| GET | `/api/tenants/:tenantId/dashboard-layouts/:layoutId` | `GetLayout` | Get layout by ID |
| PUT | `/api/tenants/:tenantId/dashboard-layouts/:layoutId` | `UpdateLayout` | Replace layout name + widgets |
| DELETE | `/api/tenants/:tenantId/dashboard-layouts/:layoutId` | `DeleteLayout` | Soft-delete layout |

## II. Implementation Phases

### Phase 1 — Database Migration

**Goal**: Create `dashboard_layouts` table with JSONB widgets and business-rule indexes.

**Files**:
- `migrations/000009_create_dashboard_layouts_table.up.sql`
- `migrations/000009_create_dashboard_layouts_table.down.sql`
- `migrations/000011_add_user_id_to_dashboard_layouts.up.sql`
- `migrations/000011_add_user_id_to_dashboard_layouts.down.sql`

**Key elements**:
- UUID PK with `gen_random_uuid()`
- `tenant_id` FK → `tenants(id) ON DELETE CASCADE`
- `widgets JSONB NOT NULL DEFAULT '[]'`
- Partial unique index `(tenant_id, name) WHERE deleted_at IS NULL`
- Tenant-scoped list index `(tenant_id) WHERE deleted_at IS NULL`
- `set_dashboard_layouts_updated_at` trigger (reuses existing `update_updated_at_column()` function)

---

### Phase 4 — Application Service Layer

**Goal**: Orchestrate domain + repository; enforce business rules.

**Files**:
- `internal/app/dashboard_layouts/service.go`

**Operations**:
1. `ListLayouts(ctx, tenantID)` → `([]*DashboardLayout, error)`
2. `GetLayout(ctx, tenantID, layoutID)` → `(*DashboardLayout, error)` — returns `ErrLayoutNotFound` if not found
3. `CreateLayout(ctx, tenantID, cmd)` → `(*DashboardLayout, error)`
   - Count check (< 3) → `ErrLimitReached` if exceeded
   - Name uniqueness check → `ErrDuplicateName` if conflict
   - Both checks + insert in a transaction
4. `UpdateLayout(ctx, tenantID, layoutID, cmd)` → `(*DashboardLayout, error)`
   - Existence check → `ErrLayoutNotFound`
   - Name uniqueness check (excluding self) → `ErrDuplicateName`
5. `DeleteLayout(ctx, tenantID, layoutID)` → `error`
   - Existence check → `ErrLayoutNotFound`
   - Count check (> 1) → `ErrCannotDeleteLastLayout`
   - Soft delete

**Response envelope** (matching Pact):
```json
{ "success": true, "data": { ... } }
{ "success": true, "data": [...], "meta": { "total": N, "limit": 3 } }
{ "success": true, "message": "Layout eliminado correctamente" }
{ "success": false, "error": "..." }
```

**Error mapping**:

| Domain Error | HTTP Status | error field |
|---|---|---|
| `ErrLayoutNotFound` | 404 | `"Layout no encontrado"` (GET/DELETE) or `"NOT_FOUND"` (PUT) |
| `ErrDuplicateName` | 409 | `"DUPLICATE_NAME"` |
| `ErrLimitReached` | 403 | `"LIMIT_REACHED"` |
| `ErrCannotDeleteLastLayout` | 400 | `"No se puede eliminar el único layout"` |
| Unauthenticated | 401 | `"No autorizado"` (handled by middleware) |
